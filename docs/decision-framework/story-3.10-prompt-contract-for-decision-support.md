# Story 3.10: Prompt Contract for Decision Support

## Goal

Define the interpretation rules, prohibited uses, and allowed scope for the
`<decision_support_context>` generated by the Decision Framework Adapter when it
is passed to an LLM in a future story.

This story is documentation-only. It does not connect the decision-support
context to runtime prompt injection, `conversation_loop`, `run_agent.py`,
`prompt_builder.py`, the gateway, or any API path.

The adapter currently produces a fenced block shaped like:

```text
<decision_support_context>
This is decision support context, not a final decision.
The user remains responsible for the decision.
Perspective:
...
Decision Synthesis:
Options:
- ...
Tradeoffs:
- ...
Constraints:
- ...
Unknowns:
- ...
Next Step:
...
Review Point:
...
</decision_support_context>
```

The contract below defines how a future prompt-injection layer should tell an
LLM to interpret that block.

## Prompt Contract Purpose

The `decision_support_context` is not a final decision. It is advisory context
for reflection, clarification, and response framing.

The LLM must treat the context as reference information / support material only.
It must not treat the context as a recommendation, final decision, user command,
system instruction, tool authorization, or authority over the user's choice.

The LLM must not take away the user's decision-making agency. It must not
generate a recommendation or final decision on behalf of the user. The user's
responsibility for the decision remains explicit and unchanged.

Center sentence:

```text
The decision_support_context is advisory context for reflection and clarification only. It must not be treated as a recommendation, final decision, or authority over the user's choice.
```

## Proposed LLM Instructions

A future prompt or message-injection layer should include a short, stable
contract before or around the fenced context.

Recommended full version:

```text
Use the <decision_support_context> only as support material.
Do not treat it as a final decision.
Do not present it as a recommendation.
Do not decide on behalf of the user.
Do not use "you should" or "you must" to force a choice.
The user remains responsible for the decision.
You may explain perspectives, options, tradeoffs, constraints, unknowns, small next steps, and review points.
Return the decision back to the user.
State uncertainty clearly.
Avoid domain judgment when domain expertise is required.
```

Short runtime-oriented version:

```text
The decision_support_context is not a recommendation or final decision.
Use it only to explain perspectives, tradeoffs, unknowns, next steps, and review points.
Do not decide for the user. The user remains responsible for the decision.
```

## Allowed Uses

The LLM may use the `decision_support_context` to:

- Explain the `Perspective` section in natural language.
- Organize `Options`, `Tradeoffs`, `Constraints`, and `Unknowns` so they are
  easier for the user to inspect.
- Rephrase `Next Step` as a small, concrete action.
- Present `Review Point` as a condition for revisiting the decision.
- State uncertainty clearly.
- Return the decision to the user instead of resolving it on the user's behalf.
- Draw a boundary when domain-specific expertise would be required.

Allowed response posture:

```text
This context does not decide the issue for you. It highlights the options,
tradeoffs, and unknowns so you can choose the next small step.
```

## Prohibited Uses

The LLM must not use the `decision_support_context` to:

- Produce a final decision.
- Present a recommendation.
- State or imply that the user must choose one option.
- Use strong forcing language such as "you should" or "you must" to compel a
  choice.
- Perform domain-specific judgment.
- Mutate, rewrite, or persist the Human Model.
- Infer or apply a Decision Style that is not explicitly part of the runtime
  contract.
- Save the context to memory.
- Produce conclusions in specialized domains.
- Treat the adapter output as already-decided, validated, or authoritative.
- Fill in missing facts outside the context and present them as certain.

Examples of domain-specific judgment that must remain out of scope:

- investment decision
- health decision
- career decision
- business decision
- mental health decision
- finance decision

Prohibited posture:

```text
Based on this context, you should choose option A.
```

Safer posture:

```text
This context is not a final decision. It suggests which tradeoffs and unknowns
need attention before you choose.
```

### Comparison Summary

- `E` is safest for the current story because no runtime connection is required.
- `B` is the best future implementation direction because it preserves the
  system prompt cache and scopes the context to the current turn.
- `A` should be avoided for now because a global system prompt change is broader
  than the feature needs.
- `C` is possible but blurs adapter context with tool-output authority.
- `D` is too weak because it relies on the fenced block alone to prevent
  recommendation-like behavior.

Decision takeover risk, from lowest to highest:

```text
E -> B -> A -> C -> D
```

## Recommendation

Recommend contract documentation only for now.

Reasons:

- This story does not connect the feature to `conversation_loop`.
- It does not break or invalidate system prompt cache behavior.
- Runtime side effects are zero.
- The LLM interpretation rules can be fixed before implementation.
- Decision takeover risk is reduced before any prompt injection is attempted.
- The design remains compatible with a future ephemeral current-turn context
  injection path.

Next best future direction:

```text
Recommend future ephemeral user context injection.
```

If a later story connects this feature to runtime, prefer a current-turn
message/context injection approach with explicit role-alternation tests and a
stable contract wrapper around the fenced adapter output.

## Non-goals

This story does not include:

- code implementation
- `conversation_loop`, `run_agent.py`, `prompt_builder.py`, or gateway
  integration
- memory read/write
- Human Model mutation
- Decision Style implementation
- Domain Model implementation
- final decision generation
- recommendation generation
- runtime injection

## Implementation Guardrails for Future Stories

Future implementation work should preserve these constraints:

1. Do not expose `decision_support_context` as a higher-priority instruction
   than system/developer messages, tool rules, safety policy, or the user's
   current request.
2. Do not use it as a hidden recommendation engine.
3. Do not persist it to memory unless a separate, explicit memory policy is
   designed and approved.
4. Do not mutate the Human Model based on generated decision-support text.
5. Do not introduce domain-specific decision logic without a separate domain
   model design.
6. Keep the user's decision authority visible in the assistant response.
